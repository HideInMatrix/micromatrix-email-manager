# Requirements: daisyUI 工作台信息架构修正

## 问题

当前界面已经切换到 daisyUI 风格，但页面职责和配置流程仍有几处不符合实际使用习惯：邮件工作台被表述成后台，OAuth 配置缺少“创建配置”的过程，连接邮箱入口放错页面，本地处理规则也不能按邮箱类型区分。同时，API Token 的身份模型需要明确区分“用户账号”和“邮箱账号”，避免把登录用户、OAuth 邮箱连接和对外接口授权混为一谈。

## 范围

- 调整设计文档，使它成为可复用的界面规范，不包含一次性的目的声明。
- 明确 `/` 是前台邮件工作台，`/dashboard/*` 是后台管理页。
- 保留未登录访问后台页面时跳转 `/login` 的行为。
- 重构 OAuth 配置页：先选择邮箱类型，再填写该类型需要的配置，保存后以卡片展示。
- 重构邮箱账号页：连接 Gmail / Outlook 的入口放在账号页，并根据 OAuth 配置完成度启用或禁用。
- 为本地邮件处理规则增加邮箱类型字段，使不同类型邮箱可以创建独立规则。
- 明确 API Token 绑定用户账号，不绑定邮箱账号；对外接口通过请求参数指定目标邮箱账号或收件邮箱。

## 用户故事

1. 作为使用者，我希望打开 `/` 时看到的是邮件处理工作台，而不是后台管理页面。
2. 作为管理员，我希望未登录访问后台配置页面时被跳转到登录页，避免管理入口裸露。
3. 作为管理员，我希望 OAuth 配置有清晰的创建流程，先选邮箱类型，再填对应字段。
4. 作为管理员，我希望保存后的 Gmail / Outlook 配置以卡片展示，便于查看状态和继续编辑。
5. 作为管理员，我希望在邮箱账号页面连接多个邮箱账号，并且只有对应 OAuth 配置完成后才能连接。
6. 作为管理员，我希望 Gmail 和 Outlook 可以拥有不同的本地处理规则。
7. 作为普通用户，我希望创建 API Token 时自动绑定当前登录用户账号，而不是选择某个邮箱账号。
8. 作为管理员，我希望代用户创建 API Token 时选择的是用户账号，而不是该用户连接的邮箱账号。
9. 作为外部系统调用方，我希望携带 API Token 后，通过请求参数指定要查询或处理的邮箱账号、收件邮箱，并且只能访问该 Token 用户有权限的邮件数据。

## 验收标准

### R1. 设计文档可复用

- 当阅读 `specs/ui-redesign/design.md` 时，系统应只描述可复用的风格、布局、组件和交互规则。
- 当设计文档描述页面边界时，系统应明确区分前台邮件工作台和后台管理页。
- 当 specs 记录业务规则、权限模型或接口语义时，系统应将其放在 `requirements.md` 或 `tasks.md`，不应放入 `design.md`。
- 当维护 `specs/ui-redesign/design.md` 时，系统应将其视为 UI 设计稿提示词，而不是需求分析或业务逻辑文档。

### R2. 页面访问边界

- 当用户访问 `/` 时，系统应显示邮件工作台，并避免将其称作后台页面。
- 当未登录用户访问 `/dashboard`、`/dashboard/accounts`、`/dashboard/config` 或 `/dashboard/events` 时，系统应跳转到 `/login?redirect=...`。
- 当管理员查看后台页面左侧菜单时，系统不应展示首页/邮件工作台入口。
- 当前台和后台页面渲染时，系统应通过 Nuxt layout 区分页面外壳，而不是在每个页面里重复实现布局结构。

### R3. OAuth 配置创建流程

- 当管理员打开 `/dashboard/config` 时，系统应提供创建 OAuth 配置的步骤。
- 当管理员选择 Gmail 时，系统应显示 Gmail 所需字段，包括 Pub/Sub Topic。
- 当管理员选择 Outlook 时，系统应只显示 Outlook 所需字段。
- 当配置保存成功后，系统应以卡片形式展示已保存的配置状态、回调地址和能力信息。

### R4. 邮箱账号连接入口

- 当管理员打开 `/dashboard/accounts` 时，系统应看到 Gmail / Outlook 的连接入口。
- 当某个邮箱类型的 OAuth 配置未完成时，系统应禁用该类型的连接按钮并提示先完成配置。
- 当 OAuth 配置已完成且该 provider 支持 OAuth 时，系统应允许从账号页发起连接。
- 当管理员连接、同步、监听或删除账号时，系统应保留现有接口行为。

### R5. 按邮箱类型创建规则

- 当管理员创建规则时，系统应要求选择 Gmail 或 Outlook。
- 当系统保存规则时，数据库应记录该规则所属的邮箱类型。
- 当系统应用规则时，规则应只匹配同一邮箱类型的邮件。
- 当系统展示规则时，界面应能看出每条规则属于哪个邮箱类型。

### R6. daisyUI 组件约束

- 当实现表单、按钮、卡片、表格、徽标、步骤和导航时，系统应优先使用 daisyUI 组件类。
- 当 daisyUI 已有相同语义组件时，系统不应创建重复的本地 UI 组件。
- 当页面展示文本、邮箱、搜索或密钥输入时，系统应统一使用 daisyUI input field 样式，而不是散落的裸输入框。
- 当用户输入登录密码、注册确认密码或 Client Secret 时，系统应提供眼睛按钮用于显示或隐藏输入内容。
- 当用户执行删除、移入垃圾箱或撤销 Token 等破坏性操作时，系统应使用 daisyUI modal 确认框，而不是浏览器原生 `window.confirm`。

### R7. 验证

- 当实现完成后，系统应通过项目构建。
- 当可以本地运行时，系统应 smoke-test `/`、`/login` 和后台路由跳转行为。
- 当 API Token 身份模型实现后，系统应 smoke-test 普通用户创建 Token、管理员代用户创建 Token、外部接口按 `accountId` 或 `recipientEmail` 过滤的访问控制。

### R8. 用户账号、邮箱账号和 API Token 边界

- 当系统展示或实现账号相关功能时，系统应明确“用户账号”是登录身份，“邮箱账号”是通过 Gmail / Outlook OAuth 连接的邮件账户，两者不是同一个概念。
- 当普通用户创建 API Token 时，系统应将 Token 绑定到当前登录用户账号，不应要求或允许选择邮箱账号。
- 当管理员代用户创建 API Token 时，系统应允许选择目标用户账号，不应通过选择邮箱账号来决定 Token 归属。
- 当外部接口收到 Bearer API Token 时，系统应以该 Token 绑定的用户账号作为调用身份。
- 当外部接口请求包含 `accountId` 时，系统应只允许访问该 Token 用户账号有权限的邮箱账号；无权限时应拒绝请求。
- 当外部接口请求包含 `recipientEmail` 或 `to` 时，系统应只在该 Token 用户账号有权限的邮箱账号范围内过滤收件邮箱。
- 当外部接口请求未指定 `accountId` 时，系统应默认只返回或处理该 Token 用户账号有权限的邮箱账号数据。
- 当页面或接口文档描述 API Token 时，系统不应把 Token 表述为某个 Gmail / Outlook 邮箱账号生成的 Token。

## 约束

- 不改变 OAuth 回调协议、Token 加密方式和登录 session 语义。
- 可以修改数据库结构以支持新的业务字段。
- API Token 鉴权边界必须基于用户账号；邮箱账号只能作为请求过滤或操作目标。
- 邮箱账号必须保留用户归属关系，例如 `ownerEmail` 或等价字段，用于校验 Token 用户是否可访问该邮箱账号。
- 保留 Tailwind v4 + daisyUI 的实现方式，不回到大段自定义 CSS。
- 保留现有脏工作区，不回滚用户已有改动。

## 非目标

- 不实现 Outlook OAuth 的完整后端能力，除非已有 provider 能力支持。
- 不新增营销落地页。
- 不重写邮件同步、Webhook 或 Gmail API 逻辑。
- 不让 API Token 直接替代 Gmail / Outlook OAuth Token；API Token 只表示本系统用户身份和对外接口访问凭证。
